# Memory Management

This document defines the default rules for managing each Agent's personal long-term memory.

## 1. Scope

- This document is the team's default rule set.
- If a repository's `AGENTS.md` defines an exception, follow that repository's `AGENTS.md`.
- This document applies to each Agent's personal long-term memory.
- This document does not directly govern team repository assets such as shared rules, learning records, or templates.

## 2. Core Principles

- Personal memory must be managed, not only accumulated.
- An Agent's memory should support the Agent's role, identity, judgment, and execution quality.
- Irrelevant, outdated, misleading, or noisy memory should not be kept without review.
- Important memory may need stronger emphasis, not just passive storage.
- Every team should assign a `Memory Manager`.

## 3. Memory Manager

- The `Memory Manager` is responsible for reviewing the quality of each Agent's personal long-term memory.
- The `Memory Manager` should judge whether a memory helps the Agent act in line with the Agent's role and responsibilities.
- The `Memory Manager` should identify memory that is useless, misleading, stale, contradictory, or distracting.
- The `Memory Manager` should identify memory that is especially important and should be emphasized.
- The Agent whose memory is being reviewed should cooperate with the review and accept necessary memory updates.

## 4. Review Criteria

- A memory should be kept if it fits the Agent's identity and helps the Agent perform better.
- A memory should be deleted if it is useless noise, stale information, low-value clutter, or something that creates confusion.
- A memory should be emphasized if it is critical to the Agent's role, repeatedly relevant, or necessary to avoid important mistakes.
- A memory that conflicts with the Agent's current role, current rules, or current reality should be reviewed immediately.

## 5. Review Actions

- `Keep`: The memory is valid, useful, and aligned with the Agent's role.
- `Delete`: The memory is not worth keeping and should be removed.
- `Emphasize`: The memory is highly important and should be made more prominent than ordinary memory.
- A review decision should be based on usefulness, accuracy, and alignment with the Agent's identity.

## 6. Review Triggers

- Memory review should happen after a meaningful mistake, process failure, or task-delivery problem.
- Memory review should happen when the same type of problem appears repeatedly.
- Memory review should happen when an Agent becomes blocked because of confusion, contradiction, or noisy memory.
- Memory review should happen when the Agent's identity, role, or responsibilities change.
- Teams may also define a regular review cadence for memory maintenance.

## 7. Communication and Team Learning

- If a memory review reveals a lesson that is useful to the whole team, that lesson should be turned into team learning.
- Team-level learning should be preserved in the team's `XXXXTeam` repository when appropriate.
- A team repository may store rules, summaries, or learning outcomes from memory review.
- A team repository does not need to store every raw personal memory item.

## 8. Default Minimum Checklist

1. Identify the memory, mistake, or confusion that should be reviewed.
2. Ask the team's `Memory Manager` to review that memory.
3. Decide whether the memory should be `Keep`, `Delete`, or `Emphasize`.
4. Update the Agent's personal memory accordingly.
5. Turn any team-relevant lesson into shared team learning when needed.
